# AWS Data Engineering — Study Pack
## Tier 1 · Topic 2: Amazon S3 (Simple Storage Service)

> S3 **is** the AWS data lake — the substrate under Athena, Glue, EMR, Redshift Spectrum, Lake Formation.
> GCP analog: GCS. Apache analog: HDFS (but decoupled, not coupled).

---

## 1. What & Why
**Object storage**: objects (file + metadata) in buckets (flat namespace), accessed over HTTP by key.
Not a filesystem — no real dirs, no in-place edits, no appends. PUT / GET / DELETE.

Why does it dominate AWS DE:
- Practically infinite, **11 nines durability**, cheap.
- **Decouples storage from compute** (the lakehouse thesis): one copy in S3, many engines read independently.
- The **integration hub** — nearly every AWS data service reads/writes S3 natively.

---

## 2. Core Concepts
- **Bucket** — top container; name **globally unique**, tied to a **region** (data lives there).
- **Object** — key (full name) + value + metadata + version ID + ETag. Max 5 TB; multipart upload >~100 MB.
- **Key & prefix illusion** — namespace is **flat**; "folders" are console grouping on `/`. But prefixes are
  **real and load-bearing**: Hive-style `key=value/` prefixes drive partition pruning. Design them deliberately.
- **Consistency** — since Dec 2020, **strong read-after-write** for all ops (PUT, overwrite, DELETE, LIST).
  *History (still asked):* used to be eventually consistent for overwrites/lists; now strong.
- **Durability vs availability** — durability via auto-replication across ≥3 AZs; availability = SLA, varies by class.

---

## 3. Storage Classes (guaranteed interview topic)
| Class | Use when | Notes |
|---|---|---|
| **Standard** | Hot, frequent | Default; highest storage cost, no retrieval fee |
| **Intelligent-Tiering** | Unknown/changing access | Auto-tiers; small monitoring fee; safe lake default |
| **Standard-IA / One Zone-IA** | Infrequent, fast access | Cheaper storage + **retrieval fee**; One Zone = 1 AZ |
| **Glacier Instant Retrieval** | Archive, rare, ms latency | Cheap storage, higher retrieval |
| **Glacier Flexible / Deep Archive** | Compliance, cold | Cheapest; retrieval minutes–hours |

**Lifecycle policies** automate transitions (IA @30d → Glacier @90d → expire @7y). Core cost hygiene.

---

## 4. Security (4 layers)
- **IAM policies** (identity-based) — what a principal can do.
- **Bucket policies** (resource-based) — cross-account, TLS enforcement, etc.
- **Block Public Access** — override switch to prevent public exposure. **Leave ON.** (Top breach cause.)
- **Encryption:** SSE-S3 (AWS keys, default) · **SSE-KMS** (KMS keys + audit; readers need `kms:Decrypt`
  AND key policy must allow them — #1 Glue AccessDenied cause) · SSE-C/client-side (rare).
  Enforce in-transit TLS via bucket policy denying non-TLS.

---

## 5. DE-Relevant Features
- **Versioning** — keeps every version; protects overwrites/deletes; pairs with lifecycle to expire old versions.
- **Event Notifications** — on create/delete → **Lambda / SQS / SNS / EventBridge** = event-driven ingestion.
  (GCS analog: GCS → Pub/Sub.)
- **S3 Select** — SQL against a *single* object (CSV/JSON/Parquet), pull only needed rows/cols. Multi-file → Athena.
- **Access Points / Multi-Region Access Points / Requester Pays** — multi-team/account scale.
- **Replication (CRR/SRR)** — cross/same-region async copy for DR/compliance.

---

## 6. Apache / GCP Mapping
| Concept | HDFS | GCS | S3 |
|---|---|---|---|
| Model | Distributed FS | Object store | Object store |
| Storage/compute | **Coupled** | Decoupled | **Decoupled** |
| Namespace | Hierarchy | Flat + prefix | Flat + prefix |
| Mutability | Append-capable | Immutable | Immutable |
| Scheme | `hdfs://` | `gs://` | `s3://` / `s3a://` (Spark) |
| Events | — | GCS→Pub/Sub | S3→Lambda/SQS/SNS/EventBridge |

**Interview talking point:** HDFS→S3 = coupled→decoupled. Lose data locality, gain independent scaling,
lower cost, many engines on one copy. Spark uses the **`s3a://`** connector.

---

## 7. Performance & Best Practices
- **Partition** with Hive-style prefixes (`dt=2026-06-29/`) for pruning. Bad partitioning → full scans.
- **File size** — the **small-files problem** applies. Target **~128 MB–1 GB**; compact small files.
- **Parquet/ORC + Snappy/ZSTD**, not CSV/JSON → columnar pruning, smaller, cheaper Athena scans.
- **Request scaling** — 3,500 writes / 5,500 reads per sec **per prefix**; spread across prefixes at extreme scale.
- **Multipart upload** for large objects; **Transfer Acceleration** for long-haul.
- **Lifecycle policies** for cost; **Intelligent-Tiering** for unpredictable access.
- **VPC Gateway Endpoint for S3** — keeps traffic off public internet, avoids NAT cost.

---

## 8. Common Mistakes & Limits
- **Small files** everywhere (the classic killer).
- **Treating it as a filesystem** — renames = copy+delete every object. Use **S3A/job committers** in Spark.
- **CSV in the lake** → huge Athena bills.
- **Forgetting KMS perms** → AccessDenied.
- **Block Public Access off** → breach.
- **No lifecycle policy** → cost creep.
- **Over-partitioning** on high-cardinality column → millions of tiny partitions.

---

## ⭐ Key Takeaways
1. S3 = the lake; object store, flat namespace, immutable objects, strong consistency.
2. **Decoupled storage/compute** is the whole point — many engines, one copy.
3. Storage classes + lifecycle = cost control; Intelligent-Tiering is the safe default.
4. Security: IAM + bucket policy + Block Public Access + encryption (mind **KMS perms**).
5. Performance = **Parquet + right file size + good partitioning**; avoid small files.
6. Event notifications enable event-driven ingestion; `s3a://` is the Spark connector.

### Q1. "Object storage, not a filesystem" — and one pipeline consequence
Objects (file + metadata) in buckets (flat namespace), accessed over HTTP by key. No real dirs,
no in-place edits, no appends — PUT / GET / DELETE.
**Consequence:** objects are **immutable** — you can't update a row in place or append; a "change" rewrites
the whole object, a "rename" is copy + delete of every object. This is *why* table formats (Iceberg/Hudi/Delta)
exist and why Spark commit-on-S3 is slow. (Name the consequence, not just the definition.)

---

### Q2. "Decouples storage from compute" — vs HDFS
One copy in S3; many engines (Athena, EMR, Redshift, Glue) read independently and scale separately = lakehouse thesis.
**vs HDFS:** HDFS **couples** storage + compute — data lives on compute nodes' disks and jobs exploit **data locality**
(compute runs where data is). S3 separates them — compute reads over the network, losing locality but gaining
independent scaling and many-engines-one-copy. (Always close the comparison the question asks.)

---

### Q3. Glue AccessDenied despite s3:GetObject allowed — first hypothesis
**SSE-KMS.** Objects encrypted with a KMS key; Glue role lacks **`kms:Decrypt`** (or the **KMS key policy** doesn't
grant the role). S3 surfaces a KMS failure as `AccessDenied`. Check default encryption + key policy first.
Secondary: ARN is bucket but not `/*`; explicit Deny in SCP/bucket policy; cross-account bucket policy missing.

---

### Q4. 50,000 tiny JSON files/day — diagnose + fixes
**Diagnosis:** small-files problem + bad format. Per-object overhead (list/open, one task per file) dominates;
JSON is row-based, uncompressed, no columnar pruning → Athena scans every byte (billed per byte scanned).
**Fixes:**
1. **Compact + convert to Parquet** (Glue/Spark) into ~128 MB–1 GB files, Snappy/ZSTD.
2. **Partition** output (`dt=YYYY-MM-DD/`) so Athena prunes.
3. Bonus: repartition/coalesce upstream so the job stops emitting tiny files.

---

### Q5. Medallion lake S3 layout (daily event data)
```
s3://acme-bronze/events/dt=2026-06-29/...         # raw, as-ingested
s3://acme-silver/events/dt=2026-06-29/...         # cleaned, deduped, conformed
s3://acme-gold/events_daily_agg/dt=2026-06-29/... # business aggregates / marts
```
- **Buckets/prefix:** separate buckets per layer (clean IAM/lifecycle boundaries) or one bucket + top-level prefixes
  (call out the tradeoff). Hive-style date partitioning `dt=YYYY-MM-DD/`.
- **Format:** Bronze = raw source (JSON/CSV) or convert on land; **Silver/Gold = Parquet** (Snappy/ZSTD), ~128 MB–1 GB.
  Use **Iceberg** on Silver/Gold if upserts/mutability needed.
- **Partitioning:** by event/ingestion date; deeper only for low-cardinality, frequently-filtered cols. Avoid over-partitioning.
- **Lifecycle:** Bronze raw → Standard-IA @30d → Glacier @90d → expire at retention; Gold stays Standard (hot).
  (= GCP medallion + GCS lifecycle, 1:1.)

---

### Q6. Auto-start pipeline on file drop
**S3 Event Notifications** on landing bucket (`s3:ObjectCreated:*`, scoped to vendor prefix) →
- **Lambda** (lightweight), or
- **EventBridge** (preferred routing/filter/fan-out) → **Step Functions / Glue / Lambda**, or
- **SQS** to buffer bursty volume.
Robust pattern: S3 → EventBridge → Step Functions → Glue, SQS as spike buffer.
**GCP equivalent:** GCS Object Notifications → **Pub/Sub** → Cloud Functions / Composer / Dataflow.

---

### Q7. 500 TB lake cost spiking — levers + tradeoffs
- **Storage class/lifecycle:** cold → Standard-IA → Glacier; Intelligent-Tiering for unpredictable.
  *Tradeoff:* retrieval fees + latency; wrong on hot data costs more.
- **Format:** CSV/JSON → Parquet/ORC + compression. *Tradeoff:* one-time rewrite cost; big ongoing scan/storage savings.
- **File sizing/compaction:** small files → ~128 MB–1 GB. *Tradeoff:* compaction compute cost.
- **Partitioning:** add/repair for pruning; avoid over-partitioning. *Tradeoff:* re-layout cost; high-cardinality backfires.
- **Query-side:** Athena bills per byte scanned — projection, partition predicates, CTAS/materialized Gold aggregates.
  *Tradeoff:* materialization adds storage + refresh.
- **Versioning hygiene:** expire old versions + incomplete multipart uploads. *Tradeoff:* lose deep history.
- **Replication/egress:** drop unneeded CRR; VPC S3 Gateway Endpoint to cut NAT/egress. *Tradeoff:* weaker DR.
- **Order of attack:** format + compaction + partitioning (biggest, lowest-risk) → tiering → query-side.
